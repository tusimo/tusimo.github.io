面试题

描述下乐观锁和悲观锁

我们在面试的时候经常遇到面试官会问到乐观锁和悲观锁，本文通过图文进行分类描述。首先乐观锁和悲观锁皆为抽象对象，并没有这两个真实的锁。

乐观锁

顾名思义乐观锁是一种很乐观的锁这不是废话吗。何为乐观呢？我们总是假设最好的情况，认为我们在获取数据的时候，别人不会去修改数据。只有在我们修改数据的时候去判断下这个数据是否已经被修改了，没有修改的情况下，我们正常修改数据。
下面我们来用mysql实现下这个乐观锁.
假设我们有下面的数据库表products。

id name stock
1 旺仔 10
2 牛奶 20

让我们来用乐观锁实现这个商品库存修改

我们先查询数据，并保存当时的库存为stock_num
select * from prdocts where id = 1; 让我们更新库存减1
update products set stock = 9 where id = 1 and stock = 10;

如上所示，我们在查询的时候并未加锁，而是在更新的时候判断数据是否被修改过。常用的判断方法：版本号 时间戳 CAS。CAS其实就是Compare And Swap，即比较并交换。我们当前使用的类似版本号，也就说如果数据是读取的时候是一样的，我们才会更新，否则的话不更新。这个就是乐观锁。

mysql数据库并没有乐观锁的实现，需要我们在开发的时候自己编写。乐观锁只是一个抽象的锁定义。

但是乐观锁有一个问题，就是ABA问题，什么叫ABA问题呢？顾名思义，一个数据原本是A,那么乐观锁是怎么判断的呢？我们更新的时候发现A还是原值A那么我们认为没有改变过A。那么如果这个A是改到B后又改到A的情况就不对了。这就是ABA问题。

乐观锁在读操作较多的情况还是比较合适的。

悲观锁

下面我们讲一讲悲观锁，同样悲观锁是一种抽象的定义，那么悲观锁可以分为共享锁 排它锁，也就是我们常说的读锁和写锁。那么我们分别讲一讲这两种锁机制。

前提：要使用悲观锁，我们必须关闭mysql数据库的自动提交属性，因为MySQL默认使用autocommit模式，也就是说，当你执行一个更新操作后，MySQL会立刻将结果进行提交。

共享锁（读锁，S锁）

共享锁（读锁）是允许共享读的，也就说允许多个线程同时读取相同的数据，但是阻止线程写数据。

mysql 如何使用
select * from user where id = 1 lock in share mode;

排它锁 （写锁，X锁）

排它锁 （写锁） 获得该锁的线程可以操作数据，而其他线程既不允许读也不允许写。

若事务 1 对数据对象A加上X锁，事务 1 可以读A也可以修改A，其他事务不能再对A加任何锁，直到事物 1 释放A上的锁。这保证了其他事务在事物 1 释放A上的锁之前不能再读取和修改A。排它锁会阻塞所有的排它锁和共享锁

mysql 如何使用
select * from user where id = 1 for update;

MySQL锁介绍

行锁

行锁 是指针对一行记录加锁，既可以是S锁，也可以是X锁。这里需要提一下，mysql针对数据加锁是针对索引的，无论是主键索引还是普通索引。如果无法使用索引，将锁住整个表。

表锁

表锁 顾名思义，讲锁住整个数据表。这样将大大限制了并发操作。但是表锁是线程独占的，那么不会出现死锁这种情况。

页锁

页锁 我们知道操作系统在读取数据的时候是按照页加载的，所以这个页锁是按照页进行锁数据的。这个在BDB存储引擎上会有，了解即可。

区别

存储引擎 行锁 表锁 页锁
MyISAM N Y N
BDB N Y Y
InnoDB Y Y N

表锁 开销小，加锁快；不会出现死锁(因为MyISAM会一次性获得SQL所需的全部锁)；锁定粒度大，发生锁冲突的概率最高,并发度最低。
页锁 开销和加锁速度介于表锁和行锁之间；会出现死锁；锁定粒度介于表锁和行锁之间，并发度一般
行锁 开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。
