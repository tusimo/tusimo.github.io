kafka系统介绍

名词解释

broker

处理节点，一个kafka节点认为是一个broker,一个或者多个broker可以组成一个kafka集群。

topic

消息主题，每一个发送到kafka的消息需要指定一个topic。消费的时候也需要指定消费的topic。

consumer

即消费者，从kafka集群消费消息。

producer

即生产者，负责向kafka发送消息。

consumer group

一组consumer的集合，每一个consumer属于一个consumer group.消息只会被一个consumer group中的一个consumer消费。不同consumer group 可以消费同一个消息。

partition

物理概念，topic可以分为多个partition进行存储。

消费模型

队列模型

每个消息只会被消费一次。

广播模型

每个消息会被消费多次。

面试题|memcache和redis的区别

综述

都是数据缓存服务，都支持key-value缓存数据，减少数据库查询次数。

服务方式

服务提供方式都支持TCP连接传输，都支持UDP连接传输。如果用户进程和服务进程在同一台机器还支持unix域套接字通信。

不同点

事件模型

redis基于epoll,memcache 基于libevent,libevent也是基于epoll的，所以两者都是epoll这种先进的事件模型。
redis是单线程的服务器。并不是redis只有一个主线程，而是只有一个主线程进行event loop,其他线程会处理其他的工作，比如：数据存储。

而memcache是多线程的，使用的是master-worker的方式，至县城监听端口，建立连接，然后顺序分配给每个工作线程。每个工作线程都有一个event loop.各个线程之间采用管道通信。

内存分配

memcahce 是采用内存池进行分配，会预先分配一个大内存块，可以减少内存分配次数，提高效率。
redis 没有内存池，是即用即分配，需要的时候直接向操作系统申请。

数据库的实现

memcache 只支持 key-value的数据存储。
redis 支持 string list set sorted set hashtable5种数据结构。
redis 支持多个数据库0-15。
redis 支持数据持久化。

过期策略

memcache 不是定期检查数据过期状态，而是在客户端进程使用的时候进行检查，这样可以减少额外的cpu进行过期检查，缺点是当有的数据长时间未被使用，将占据内存。
redis 不是将过期时间存储在数据结构中，而是单独使用一个dict来保存过期时间，并且也是采用memcache的惰性删除方式。但是redis有定时删除用于补救内存占用，存在一长一短的两种方式，短期是随机选取部分过期数据删除，长期的会执行一次长时间的删除。

redis持久化

RDB持久化

save和bgsave命令执行时，会将数据库保存到文件里。保存的时候会将过期数据剔除掉。
bgsave 会fork出一个子进程，让子进程进行数据保存工作，写入temp文件。当子进程完成数据保存后会给父进程发送信号，父进程将temp文件修改为真正的rdb文件。但是这个其中有一个问题，在子进程保存的过程中如果有数据修改，会导致持久化的数据和当前数据库的数据不一致。

AOF持久化

该方式不是存储的数据库本身，而是存储的命令，类似于 mysql的redo log.存储每次执行的命令，将命令依次重放就是最终的结果了。

每次执行命令的时候都会存入aof_buf，在event loop循环时将aof_buf 存入aof文件。

同时提供 aof_rewrite将现有的数据库生成对应的命令，然后把命令存入aof文件。

aof也支持后台模式，那么也会遇到rdb的问题，父进程数据更新了怎么办？这个时候aof并不和子进程通信，而是将持久化期间的命令追加到aof文件即可。

redis事务

redis事务是指可以同时执行多个命令，使用multi命令开启。但是并不会在执行错误时进行回滚。支持watch观察变量是否被改变了。

面试题|mysql数据库事务介绍

为什么要使用事务

因为我们针对数据库的操作有些时候需要多个语句同时执行成功，并且不能被其他线程操作干扰，保证数据和预期一样。

事务的性质

原子性（A） 构成事务的的所有操作必须是一个逻辑单元，要么全部执行，要么全部不执行。
稳定性（C） 数据库在事务执行前后状态都必须是稳定的。
隔离性（I） 事务之间不会相互影响。
持久性（D） 事务执行成功后必须全部写入磁盘。

事务的级别

Read UnCommitted(未提交读)

一个事务读取到了另一个事务还未提交的数据。也就是读取到了脏数据。该级别会出现脏读，不可重复读，幻读

Read Committed(已提交读)

一个事务读取到另一个事务已经提交的数据。该级别会出现不可重复读，幻读

Repeatable Read(可重复读)

一个事务开启后可以对相同的数据进行重复读取。该级别会出现幻读

Serializable(可串行化)

该级别相当于串行执行事务。

事务中的读取数据问题

脏读

脏读就是指当一个事务正在访问数据，并且对数据进行了修改，而这种修改还没有提交到数据库中，这时，另外一个事务也访问这个数据，然后使用了这个数据。

不可重复读

是指在一个事务内，多次读同一数据。在这个事务还没有结束时，另外一个事务也访问该同一数据。那么，在第一个事务中的两次读数据之间，由于第二个事务的修改，那么第一个事务两次读到的的数据可能是不一样的。这样就发生了在一个事务内两次读到的数据是不一样的，因此称为是不可重复读。

幻读

第一个事务对一个表中的数据进行了修改，这种修改涉及到表中的全部数据行。同时，第二个事务也修改这个表中的数据，这种修改是向表中插入一行新数据。那么，以后就会发生操作第一个事务的用户发现表中还有没有修改的数据行，就好象发生了幻觉一样。

总结

我们用表格来整理下

隔离级别 脏读 不可重复读 幻读 锁
未提交读 Y Y Y 不加锁
读已提交 N Y Y 加读锁
可重复读 N N Y 加读写锁
可串行化 N N N 表级共享锁，读写阻塞

事实上mysql并不是简单的对事务加锁，为了更高的读写并发，通常使用mcvv多版本控制增加读写并发。这将涉及到redo log undo log使用。

面试题|乐观锁和悲观锁

面试题

描述下乐观锁和悲观锁

我们在面试的时候经常遇到面试官会问到乐观锁和悲观锁，本文通过图文进行分类描述。首先乐观锁和悲观锁皆为抽象对象，并没有这两个真实的锁。

乐观锁

顾名思义乐观锁是一种很乐观的锁这不是废话吗。何为乐观呢？我们总是假设最好的情况，认为我们在获取数据的时候，别人不会去修改数据。只有在我们修改数据的时候去判断下这个数据是否已经被修改了，没有修改的情况下，我们正常修改数据。
下面我们来用mysql实现下这个乐观锁.
假设我们有下面的数据库表products。

id name stock
1 旺仔 10
2 牛奶 20

让我们来用乐观锁实现这个商品库存修改

我们先查询数据，并保存当时的库存为stock_num
select * from prdocts where id = 1; 让我们更新库存减1
update products set stock = 9 where id = 1 and stock = 10;

如上所示，我们在查询的时候并未加锁，而是在更新的时候判断数据是否被修改过。常用的判断方法：版本号 时间戳 CAS。CAS其实就是Compare And Swap，即比较并交换。我们当前使用的类似版本号，也就说如果数据是读取的时候是一样的，我们才会更新，否则的话不更新。这个就是乐观锁。

mysql数据库并没有乐观锁的实现，需要我们在开发的时候自己编写。乐观锁只是一个抽象的锁定义。

但是乐观锁有一个问题，就是ABA问题，什么叫ABA问题呢？顾名思义，一个数据原本是A,那么乐观锁是怎么判断的呢？我们更新的时候发现A还是原值A那么我们认为没有改变过A。那么如果这个A是改到B后又改到A的情况就不对了。这就是ABA问题。

乐观锁在读操作较多的情况还是比较合适的。

悲观锁

下面我们讲一讲悲观锁，同样悲观锁是一种抽象的定义，那么悲观锁可以分为共享锁 排它锁，也就是我们常说的读锁和写锁。那么我们分别讲一讲这两种锁机制。

前提：要使用悲观锁，我们必须关闭mysql数据库的自动提交属性，因为MySQL默认使用autocommit模式，也就是说，当你执行一个更新操作后，MySQL会立刻将结果进行提交。

共享锁（读锁，S锁）

共享锁（读锁）是允许共享读的，也就说允许多个线程同时读取相同的数据，但是阻止线程写数据。

mysql 如何使用
select * from user where id = 1 lock in share mode;

排它锁 （写锁，X锁）

排它锁 （写锁） 获得该锁的线程可以操作数据，而其他线程既不允许读也不允许写。

若事务 1 对数据对象A加上X锁，事务 1 可以读A也可以修改A，其他事务不能再对A加任何锁，直到事物 1 释放A上的锁。这保证了其他事务在事物 1 释放A上的锁之前不能再读取和修改A。排它锁会阻塞所有的排它锁和共享锁

mysql 如何使用
select * from user where id = 1 for update;

MySQL锁介绍

行锁

行锁 是指针对一行记录加锁，既可以是S锁，也可以是X锁。这里需要提一下，mysql针对数据加锁是针对索引的，无论是主键索引还是普通索引。如果无法使用索引，将锁住整个表。

表锁

表锁 顾名思义，讲锁住整个数据表。这样将大大限制了并发操作。但是表锁是线程独占的，那么不会出现死锁这种情况。

页锁

页锁 我们知道操作系统在读取数据的时候是按照页加载的，所以这个页锁是按照页进行锁数据的。这个在BDB存储引擎上会有，了解即可。

区别

存储引擎 行锁 表锁 页锁
MyISAM N Y N
BDB N Y Y
InnoDB Y Y N

表锁 开销小，加锁快；不会出现死锁(因为MyISAM会一次性获得SQL所需的全部锁)；锁定粒度大，发生锁冲突的概率最高,并发度最低。
页锁 开销和加锁速度介于表锁和行锁之间；会出现死锁；锁定粒度介于表锁和行锁之间，并发度一般
行锁 开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低,并发度也最高。

go-micro 在调试的时候服务不自动拆除的问题

问题分析

go-micro 微服务正常运行的时候,可以自动注册和解除注册,但是再使用goland进行debug的时候注册上的服务会因为终止不能正确的解除注册.导致该服务会一直存在,并会被后续的请求负载到不能正确的响应.

解决办法

设置两个参数:RegisterInterval 表示多长时间进行重新注册,RegisterTTL表示注册过期时间,当服务崩溃了,到了过期时间会被自动拆除掉,一般设置 RegisterInterval要比 RegisterTTL 稍微短一些.
